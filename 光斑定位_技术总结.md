# 光斑定位模块 — 技术难点与创新点 (IC 设计视角)

---

## 一、核心难点

### 1.1 连通域标记 (CCL) 的单遍流式化

**问题本质：**
经典软件 CCL 算法是两遍扫描——第一遍标记所有连通域并建立等价表，第二遍合并等价标签。两遍都必须持有全帧图像，意味着需要片上帧缓存 171 MB——对 ASIC/FPGA 完全不可接受。

**解决方案：**
将 CCL 改造为**单遍流式算法**，像素逐行流入、段信息实时输出，无需帧缓存：

| 状态机 | 做的事 | 等价于软件哪步 |
|---|---|---|
| `state_cell` | 逐像素检测亮段的起点/中间/终点 | 第一遍扫描的行内游程 |
| `state_cell_process` | 段间合并 + 写入 FIFO | 等价表压缩（仅保留上一行） |
| `state_merge` | FIFO 读出上一行段，判断上下行连通 | 跨行等价标签解析 |
| `state_blk` | BRAM 管理跨切片拼接信息 | 多通道间的等价表 |

**硬件代价：** 仅需 1 行 BRAM 行缓冲 (FIFO 深度 128) 存储上一行的段信息，而非全帧 171 MB。

**IC 设计视角：** 这是经典的**算法→硬件架构重构**——将依赖全局数据的两遍算法拆分为流水线状态机，用局部缓存替代全局缓存，是数字前端设计的核心能力。

---

### 1.2 10 通道并行处理的跨切片拼接

**问题本质：**
MCDMA 将 7680×5820 图像水平切为 10 个 768×5820 的切片（注：MCDMA 10 通道按行分配，每通道 582 行）。切片边界处的光斑会被截断成两部分——上半在 CCLC[i]，下半在 CCLC[i+1]，需要跨通道缝合。

**解决方案 (merge_output + state_blk)：**

```
CCLC[i] 处理完 582 行后：
  └─ split blob 信息 (x_weight, y_weight, gray, count, start_x, length)
     写入 BRAM[merge_cnt]

merge_output 逐对比较：
  上切片 blob:  x1 = start_x,  x2 = start_x + length
  下切片 blob:  x3 = start_x_2, x4 = start_x_2 + length_2
  
  if (x1 < x4 && x3 < x2)  →  重叠 → 合并累加值 → 作为单个 blob 输出
  else                      →  无重叠 → 直接输出
```

**IC 设计视角：** 多核并行处理中的**边界数据一致性**问题——与 GPU 的 tile-based rendering、NPU 的 feature map 分区面临同类问题。merge_output 的 BRAM 存储 + 逐对比较是片内互联的低成本替代方案。

### 1.3 DSP48 流水线与像素流的时序对齐

**问题本质：**
DSP48E2 乘法器内部有 3 级流水延迟。像素数据是逐周期流入的，若不加处理，第 N 个像素的乘法结果需要等到第 N+3 周期才可用，而 `state_cell` 在第 N 周期就需要 x_weight/y_weight 值来判断段边界。

**解决方案——三级打拍匹配：**

```
pixel(8b) ─→ [data_0] ─→ [data_1] ─→ [data_2] ─→ state_cell (行内游程判断)
                  打拍1        打拍2        打拍3

col_cnt(13b) ──────────────────────────────→ DSP48 mult_x
                                             │
                                       Stage0→Stage1→Stage2
                                             3 周期后:
                                             x_weight(28b) 输出 ──│
                                                                   │
pixel 到达 state_cell ←── (data_2) 与 x_weight 同时到达 ──────────┘
```

**IC 设计视角：** 这是**流水线平衡 (pipeline balancing)** 的经典案例——通过插入等量延迟寄存器使数据路径和控制路径重新对齐，避免 DSP 成为吞吐瓶颈。面试中可进一步讨论：如果乘法器延迟变为 5 周期（如 ASIC 工艺），需要增加几级打拍。

### 1.4 反压链：4 级跨 IP 背压传播

**问题本质：**
lsp_debug 处理速度取决于光斑复杂度（复杂图像段多、FIFO 易满），不能保证永远跟上 MCDMA 的 256-bit 突发输入。若下游不提供反压，数据会丢失。

**解决方案——完整反压链路：**

```
lsp_debug/CCLC:  BRAM FIFO 满 → ready_o[i] = 0
       │
       v
Frame_Arbiter:   CCLC_ready[i] = 0 → axis_dwidth_converter 暂停 → FIFO 不读出
       │
       v
MCDMA:           MM2S_tready = 0 (对应 TDEST[i] 通道)
       │
       v
DDR:             MCDMA 停止该通道的 AXI 读请求
```

**IC 设计视角：** AXI-Stream 协议的 VALID/READY 握手在跨 IP 场景下的正确串联——每一级都必须正确传递反压，否则会出现 FIFO 溢出或死锁。这是 SoC 集成中的核心问题。

## 二、创新点 / 发光点

### 2.1 单遍流式 CCL 硬件架构 (核心创新)

传统 CCL 依赖两遍扫描 + 全局等价表，无法流式化。本设计通过**行间局部等价性**假设（上一行 FIFO + 当前行游程实时比对），将 CCL 降为单遍流式处理。代价是跨两行以上的 U 型光斑可能被拆分为多个区域——但对于光斑（通常是凸区域），此简化**不影响精度**。

**技术含量：** 对算法本质的深刻理解——知道哪些信息可以丢弃（远距离等价关系）、哪些必须保留（紧邻行连通性），从而在精度和硬件成本间取得最优折中。

### 2.2 四状态机并行流水架构

不同于常见的单状态机流水线，CCLC2_0 将功能拆分为 4 个独立状态机：

```
  state_cell         state_cell_process     state_merge         state_blk
  (游程检测)          (段合并+FIFO写)        (行间连通判断)      (跨切片BRAM管理)
      │                    │                     │                   │
      │ cells_end_pulse    │                     │                   │
      ├───────────────────→│                     │                   │
      │                    │ FIFO_wren           │                   │
      │                    ├────────────────────→│                   │
      │                    │                     │ blk_temp_flag     │
      │                    │                     ├──────────────────→│
      │                    │                     │                   │
      └── 共享: col_cnt, row_cnt, data_0/1/2/3, DSP48 输出 ────────┘
```

**IC 设计视角：** 四个状态机解耦了不同的处理阶段——游程检测关注像素边界、段合并关注段间间距、行间连通关注 FIFO 读取时序、BRAM 管理关注跨切片数据一致性。解耦后每个状态机可以独立优化时序和面积，且可以独立验证，符合模块化 IC 设计原则。

### 2.3 质心计算的逐像素流水累加

经典质心计算需要先分割出光斑区域，再对区域内所有像素做 X×gray / Y×gray 累加。本设计将分割与累加**融合为一个流水步骤**：

```
pixel 流入 → 同时做:
  ├─ 游程检测 (判断起点/中间/终点)
  ├─ mult_x: data × col_cnt  (X 质心分子累加)
  └─ mult_y: data × row_cnt  (Y 质心分子累加)
```

没有"先存全图，再算质心"的中间步骤，乘法器和累加器在每个有效像素周期都在工作，硬件利用率接近 100%。

**IC 设计视角：** 这是**计算与数据流融合**的典型设计模式——将分散的算法步骤压缩到同一个流水级，消除中间存储，降低延迟和功耗。

### 2.4 输出 80-bit 紧凑编码

每个光斑的输出数据仅 80-bit = {X_weight, Y_weight, Gray, count}，PC 端只需做一次除法得到质心。对比如果输出所有段的原始坐标列表（可能数百 bytes），80-bit 编码将 C2H 带宽需求降低约一个数量级。

**IC 设计视角：** **数据压缩在源头**——在硬件最前端做信息提取而非数据搬运，符合"靠近数据源做计算"的能效优化原则。

### 2.5 参数化架构，可复用性

所有关键参数均为 Verilog parameter，不依赖硬编码：

| 参数 | 含义 | 可扩展场景 |
|---|---|---|
| `Width` / `Height` | 图像尺寸 | 适配不同分辨率相机 |
| `merge_num` | 通道数 | 匹配不同 MCDMA 配置 |
| `X_weight_W` 等 | 累加器位宽 | 适配不同精度需求 |
| `Thres` | 噪声阈值 | 不同光照条件 |

**IC 设计视角：** 参数化 RTL 是可综合 IP 的基本要求，表明模块可以被重新实例化到不同工艺/规模的设计中。

## 三、面试应答模板

**Q: 这个模块最核心的技术挑战是什么？**

A: "最核心的挑战是将连通域标记算法（CCL）从经典的两遍扫描改造为单遍流式硬件架构。传统的 CCL 需要先扫一遍建立等价标签表，再扫一遍合并标签——这要求整帧图像缓存在内存里。我通过分析光斑的物理特性——它们是凸区域、紧邻行之间的连通关系足够确定合并——设计了一个 4 状态机的流水线架构：像素流入后，游程检测识别段边界，行间 FIFO 做等价标签解析，BRAM 做跨通道缝合。全部信息只需缓存上一行的段数据（深度仅 128），而非全帧 171 MB。这是对算法本质的深入理解后做的硬件-软件协同优化。"

**Q: 10 个 CCLC 实例并行，怎么处理边界光斑？**

A: "每个 CCLC 处理 582 行图像切片。切片边界处的光斑会被截断。我设计了一个 merge_output 模块，利用 BRAM 存储每个 CCLC 的边界 blob 信息，并对相邻切片做 'x1 < x4 且 x3 < x2' 的重叠检测。如果重叠，累加值合并为一个 blob；不重叠则独立输出。这个问题的本质是多核并行处理中的数据一致性问题，和 GPU tile-based rendering 的边界缝合是同类的。在 IC 设计中，类似的 cross-boundary data merging 也出现在 NPU 的 feature map 分区计算中。"

**Q: 你在这个设计中如何考虑 PPA？**

A: "Perfomance 方面：10 CCLC 并行是面积换吞吐的决策，在 200 MHz 下实现 20+ FPS。Area 方面：CCLC 内用 FIFO（深度 128）而非帧缓存，BRAM 用量仅 128×99 bits，比存全帧节省约 3 个数量级。Power 方面：Frame_Arbiter 的阈值过滤在数据流入口就清零了暗像素，下游 CCLC 的大部分逻辑不翻转（数据 0 天然减少组合逻辑跳变），这相当于在架构层面免费实现了 operand isolation。如果进一步优化，可以在帧间通过时钟门控关闭 CCLC 模块的时钟树。"
